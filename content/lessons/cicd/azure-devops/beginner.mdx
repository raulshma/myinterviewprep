## Your First Pipeline: The Basics ğŸ“‹

A pipeline is defined in a YAML file called `azure-pipelines.yml`:

```yaml
# azure-pipelines.yml
trigger:
  - main  # Run when code is pushed to main branch

pool:
  vmImage: 'ubuntu-latest'  # Use a Linux machine

steps:
  - task: UseDotNet@2
    displayName: 'Install .NET 8'
    inputs:
      version: '8.0.x'
  
  - task: DotNetCoreCLI@2
    displayName: 'Restore packages'
    inputs:
      command: 'restore'
  
  - task: DotNetCoreCLI@2
    displayName: 'Build'
    inputs:
      command: 'build'
      arguments: '--configuration Release'
  
  - task: DotNetCoreCLI@2
    displayName: 'Run Tests'
    inputs:
      command: 'test'
```

## Key Concepts for Beginners

### Triggers (When to Run) ğŸ¬

```yaml
trigger:
  - main        # Run on push to main
  - develop     # Also run on develop
```

### Pool (Where to Run) ğŸ’»

The pool is the machine that runs your code:

```yaml
pool:
  vmImage: 'ubuntu-latest'   # Linux
  # vmImage: 'windows-latest' # Windows
  # vmImage: 'macos-latest'   # macOS
```

### Tasks (What to Do) âœ…

Tasks are pre-built actions provided by Microsoft:

```yaml
- task: DotNetCoreCLI@2       # The task name and version
  displayName: 'Build'         # Friendly name in the UI
  inputs:                      # Configuration for the task
    command: 'build'
    projects: '**/*.csproj'
```

<KeyConcept title="The Azure DevOps Task Library">
  Azure DevOps has **hundreds of pre-built tasks** for common operations:
  - `DotNetCoreCLI@2` - .NET commands
  - `AzureWebApp@1` - Deploy to Azure
  - `Docker@2` - Build Docker images
  - `PublishTestResults@2` - Show test results
</KeyConcept>

## Classic Editor vs YAML Pipelines

Azure DevOps offers two ways to create pipelines:

| **Classic Editor** | **YAML Pipelines** |
| :----------------- | :----------------- |
| Visual drag-and-drop | Code-based definition |
| Stored in Azure DevOps | Stored in your repo |
| Easier to start | Version controlled |
| Less flexible | More powerful |

<InfoBox type="info">
  **Best Practice:** Use YAML pipelines! They're stored with your code, so you can review changes, rollback, and share across projects.
</InfoBox>

## Understanding Agents ğŸ¤–

An **agent** is the machine that runs your pipeline:

- **Microsoft-hosted agents** - Managed by Microsoft, fresh VM for each run
- **Self-hosted agents** - Your own machines, can access internal resources

```yaml

  vmImage: 'ubuntu-latest'  # Microsoft-hosted

# OR

pool:
  name: 'MyPrivatePool'     # Self-hosted pool
```
